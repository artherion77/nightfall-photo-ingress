# Staging Cloudflare Tunnel (Phase 2 C6/C7 Infra Extension)

Status: Active
Date: 2026-04-11
Owner: Systems Engineering

## 1. Purpose

Define deterministic and secure Cloudflare Tunnel integration for staging ingress reachability while preserving existing TLS termination boundaries.

Scope:
- staging container cloudflared runtime
- host-mounted local-managed Cloudflare auth material
- operational status and validation checks

Out of scope:
- storing Cloudflare credentials inside the container image or writable filesystem
- replacing Caddy as TLS terminator

## 2. Architecture

1. The staging container runs cloudflared as a managed systemd service.
2. Cloudflared authenticates in local-managed mode via auth cert + tunnel credentials JSON.
3. The auth files are mounted read-only from host paths to container paths:
   - `/home/chris/.cloudflare-secrets/npi-staging/cloudflared-auth-cert.pem` -> `/etc/cloudflared/cloudflared-auth-cert.pem`
   - `/home/chris/.cloudflare-secrets/npi-staging/tunnel-credentials.json` -> `/etc/cloudflared/tunnel-credentials.json`
4. These mounts are read-only and must remain read-only.
5. Caddy stays in-container and remains the TLS terminator for the application.
6. Cloudflare Tunnel forwards traffic to the staging endpoint; it does not terminate application TLS in this architecture.
7. WAN tunnel origin now targets a dedicated in-container Caddy listener on `https://127.0.0.1:8443` with SNI `npi.pohl-family.org`.

## 3. Authentication Model and Secret Handling

1. Authentication uses local-managed tunnel run mode:
   - `cloudflared --config /etc/cloudflared/config.yml --origincert /etc/cloudflared/cloudflared-auth-cert.pem tunnel run`
2. The config file references host-mounted auth files and is generated by `stagingctl`.
3. Auth files are host-owned and never copied into container writable layers.
4. No Cloudflare credentials may persist under container-local paths such as:
   - `/root/.cloudflared`
   - `/etc/cloudflared/*.json`
   - `/var/lib/cloudflared/*.json`
5. Validation must fail if unexpected credential artifacts are detected in container-local filesystem paths.
6. Cloudflare Origin cert and key are host-owned and mounted read-only into the container:
   - host cert: `/home/chris/.cloudflare-secrets/npi-staging/origin-cert.pem`
   - host key: `/home/chris/.cloudflare-secrets/npi-staging/origin-key.pem`
   - container cert: `/etc/caddy/tls/cloudflare-origin.crt`
   - container key: `/etc/caddy/tls/cloudflare-origin.key`

## 4. Service Supervision Model

1. Service name: `cloudflared-tunnel.service`
2. Managed by container systemd:
   - disabled by default (default-off policy)
   - restarted on failure
3. Service logs are collected via `journalctl` for troubleshooting.
4. `stagingctl cloudflared-status` is the operator status command to verify:
   - local-managed auth mount policy
   - service active state
   - tunnel connectivity signals in logs
   - recent logs tail

## 5. Default-Off Operator Controls

1. `stagingctl install` provisions cloudflared and service unit but MUST leave tunnel OFF.
2. Operators explicitly control runtime with:
   - `stagingctl cloudflared.start`
   - `stagingctl cloudflared.stop`
3. Equivalent govctl targets are available:
   - `govctl run staging.cloudflared.start --json`
   - `govctl run staging.cloudflared.stop --json`
4. Tunnel may only be started through explicit operator command surfaces.

## 6. Cloudflare Dashboard Setup (Required for Option 2)

1. Generate a Cloudflare Origin Certificate in dashboard:
   - Cloudflare Dashboard -> your zone (`pohl-family.org`) -> SSL/TLS -> Origin Server -> Create Certificate
   - Hostnames: include `npi.pohl-family.org` (and wildcard if desired)
   - Key format: PEM
2. Save files to host paths expected by stagingctl:
   - `/home/chris/.cloudflare-secrets/npi-staging/origin-cert.pem`
   - `/home/chris/.cloudflare-secrets/npi-staging/origin-key.pem`
3. Set restrictive host permissions:
   - `chmod 600 /home/chris/.cloudflare-secrets/npi-staging/origin-cert.pem`
   - `chmod 600 /home/chris/.cloudflare-secrets/npi-staging/origin-key.pem`
4. Configure Cloudflare Tunnel public hostname service for `npi.pohl-family.org`:
   - service URL: `https://127.0.0.1:8443`
   - origin server name (SNI): `npi.pohl-family.org`
   - keep TLS verification enabled.

Notes:
1. Local-managed auth material does not create/download Origin certs.
2. Certificate issuance is a dashboard/API operation requiring zone-level credentials.

## 7. DNS and TLS Implications

1. DNS points the Cloudflare tunnel hostname to Cloudflare edge as usual.
2. Inside staging, Caddy remains the TLS endpoint for the app.
3. Cloudflare Origin Certificates or TLS passthrough can be used between Cloudflare and origin, but deployment policy must preserve:
   - Caddy as the only application TLS terminator
   - no host-level TLS termination
4. Existing staging internal CA flow for LAN/operator trust remains valid and independent of cloudflared local-managed auth.

## 8. Operational Validation Checklist

1. `stagingctl create` mounts `/etc/cloudflared/cloudflared-auth-cert.pem` and `/etc/cloudflared/tunnel-credentials.json` as read-only from host secret paths.
2. `stagingctl install` verifies local-managed auth mounts, installs cloudflared when missing, and leaves `cloudflared-tunnel.service` OFF.
3. `stagingctl cloudflared.start` starts and verifies tunnel connectivity.
4. `stagingctl cloudflared.stop` stops service and verifies shutdown.
5. `stagingctl cloudflared-status --strict` succeeds with tunnel-connected log evidence after explicit start.
6. `govctl run staging.cloudflared.start --json` and `govctl run staging.cloudflared.stop --json` behave identically to stagingctl commands.
7. Caddy loads `/etc/caddy/cloudflare-origin.caddy` and serves dedicated WAN origin TLS on port `8443` when origin cert/key mounts are present.
