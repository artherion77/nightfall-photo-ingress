## 2. Path Resolution Risk Map

### Current `./metricsctl` path derivation

**DEV_LIB bootstrap (lines 18–24):**
```python
DEV_LIB = Path(__file__).resolve().parent / "dev" / "lib"
# Assumes: __file__ is at repo root
# Result: resolves to <repo_root>/dev/lib
```

**`ensure_venv()` in `dev/lib/venv_bootstrap.py` (lines 35–36):**
```python
script = Path(script_path).resolve()
repo_root = script.parent   # <-- assumes script.parent IS the repo root
venv_python = repo_root / venv_name / "bin" / "python"
# Result: looks for <script.parent>/.venv/bin/python
```

**`_repo_root()` function (lines 75–77):**
```python
def _repo_root() -> Path:
    return Path(__file__).resolve().parent
    # Result: returns repo root ONLY if metricsctl is at repo root
```

### Paths broken if `./metricsctl` were moved to `dev/lib/metricsctl.py`

| Path derivation in `_repo_root()` callers | Current resolution | If moved to `dev/lib/` | Breaks? |
|-------------------------------------------|--------------------|------------------------|---------|
| `root / "artifacts" / "metrics" / "latest"` | `<repo>/artifacts/metrics/latest` | `<repo>/dev/lib/artifacts/metrics/latest` | **YES** |
| `root / "metrics" / "runner"` | `<repo>/metrics/runner` | `<repo>/dev/lib/metrics/runner` | **YES** |
| `root / "metrics" / "systemd"` | `<repo>/metrics/systemd` | `<repo>/dev/lib/metrics/systemd` | **YES** |
| `root / "metrics" / "state"` | `<repo>/metrics/state` | `<repo>/dev/lib/metrics/state` | **YES** |
| `root / "metrics" / "output"` | `<repo>/metrics/output` | `<repo>/dev/lib/metrics/output` | **YES** |
| `root / "dev" / "bin" / "devctl"` (in `_devctl_dashboard_drift_preflight`) | `<repo>/dev/bin/devctl` | `<repo>/dev/lib/dev/bin/devctl` | **YES** |
| `.venv` lookup in `venv_bootstrap` | `<repo>/.venv` | `<repo>/dev/lib/.venv` | **YES** |
| `DEV_LIB` self-reference computation | `<repo>/dev/lib` | `<repo>/dev/lib/dev/lib` (non-existent) | **YES** |

All 8 critical path resolutions break on a naive file move.

### Required refactors for location-independence

1. **Decouple `_repo_root()` from `__file__`:** Introduce `find_repo_root(sentinel="pyproject.toml")`
   in `dev/lib/` that walks up from `Path.cwd()` until the sentinel file is found.

2. **Decouple `ensure_venv()` from `script.parent`:** Change `venv_bootstrap.ensure_venv()` to
   accept an explicit `repo_root: Path | None = None` parameter; when provided, use it directly
   instead of deriving from `script.parent`.

3. **Make `DEV_LIB` resolution independent:** Replace hard-coded
   `Path(__file__).resolve().parent / "dev" / "lib"` with a `find_repo_root()` call once that
   helper exists.

### Key compatibility observations

1. **Verb rename:** `metrics-runner publish-github` → `metricsctl publish` — the verb
   `publish-github` does not exist in `metricsctl`; operators invoking it directly against
   `metricsctl` get a parse error.

2. **Verb rename:** `metrics-runner run` → `metricsctl run-now` — similarly, `run` is not
   a `metricsctl` subcommand.

3. **Flag pass-through:** `metrics-runner run --max-retries 2 --timeout-seconds 2400` correctly
   forwards both flags via `exec "$METRICSCTL" run-now "$@"`.

4. **Deprecatable behaviors:** `publish-github` and `run` verbs (both are cosmetic aliases);
   operator documentation should direct users to `metricsctl run-now` and `metricsctl publish`.

5. **Must preserve during migration:** All three `metrics-runner` verbs must remain functional
   until removal is formally announced and documented, with a transition window.

## 5. 3-Phase Migration Plan

---

### Phase 1 — Preparation & Compatibility Layer

**Goals:**
- Make path resolution in `./metricsctl` explicit and independent of script location
- Standardize all callers to use a single canonical entrypoint (`./dev/bin/metricsctl`)
- Define compatibility shim contract for `metrics-runner`
- Add regression coverage for path resolution and caller routing

**Non-goals:**
- Do not move any files
- Do not remove `metrics-runner`
- Do not change any operator-visible CLI surface

**Preconditions:**
- This analysis document approved by maintainer
- No outstanding govctl target or MCP mapping failures
- Existing test suite passes

**Work items:**
1. Add `find_repo_root(sentinel: str = "pyproject.toml") -> Path` to `dev/lib/` — walks up
   from `Path.cwd()` until sentinel file is found; raises `RuntimeError` if not found within
   depth limit (default: 12 levels).
2. Add `repo_root: Path | None = None` parameter to `venv_bootstrap.ensure_venv()`;
   when provided, use it directly instead of deriving from `script.parent`.
3. Update `./metricsctl` `_repo_root()` to call `find_repo_root()` instead of
   `Path(__file__).resolve().parent`.
4. Update `./metricsctl` `DEV_LIB` computation to use `find_repo_root()` result.
5. Update all 9 `govctl-targets.yaml` `metrics.*` targets to invoke `./dev/bin/metricsctl`
   instead of `./metricsctl`.
6. Add comment to `dev/bin/metrics-runner` noting it is a compatibility shim over
   `dev/bin/metricsctl` and will eventually be deprecated.

**Operator-visible behaviors to remain unchanged:**
- `metricsctl status`, `run-now`, `publish`, `install`, `stop` outputs unchanged
- `metrics-runner run`, `status`, `publish-github` continue to work
- MCP task mappings produce identical results

**Minimal test updates required in this phase:**
- Add test for `find_repo_root()` sentinel-based search (found, not found, depth-exceeded)
- Add test for `ensure_venv()` with explicit `repo_root` parameter
- Add test that `metricsctl._repo_root()` returns correct value when CWD ≠ repo root

**Acceptance criteria:**
- All govctl `metrics.full` group targets pass
- `./dev/bin/metricsctl paths` works from any working directory in the repo
- `./dev/bin/metrics-runner run/status/publish-github` still delegates correctly
- MCP metrics task tests pass (`test_mcp_metrics_tasks.py`)
- Systemd unit generation produces correct `ExecStart` path
- No test regressions

**Risks:**
- `find_repo_root()` sentinel search is slow on deep/deeply-nested workspaces
  → Mitigate: cap walk depth; cache result in module-level variable
- CWD-based root discovery fails if script is invoked from outside the repo
  → Mitigate: fall back to `Path(__file__).resolve().parent` if sentinel not found within depth limit;
    log a warning

### Phase 2 — Consolidation & Refactor

**Goals:**
- Move `./metricsctl` Python implementation to `./dev/lib/metricsctl.py`
- Update `./dev/bin/metricsctl` to be a thin Python/bash launcher of `dev/lib/metricsctl.py`
- Convert `./metricsctl` at repo root to a deprecation-warning shim
- Update all caller documentation to point to `./dev/bin/metricsctl` exclusively

**Non-goals:**
- Do not remove `metrics-runner` yet
- Do not change any operator-visible CLI surface
- Do not restructure `metrics/runner/` modules

**Preconditions:**
- Phase 1 complete and all acceptance criteria validated
- `find_repo_root()` and decoupled `ensure_venv()` in production use
- No open regressions on govctl targets or MCP task tests

**Work items:**
1. Create `./dev/lib/metricsctl.py` containing all `cmd_*` functions, `build_parser()`,
   and `main()` from `./metricsctl` (adjusted imports; no functional changes).
2. Update `./dev/bin/metricsctl` (bash) to a thin launcher:
   ```bash
   exec "$(cd "$(dirname "$0")/../.." && pwd)/.venv/bin/python" \
        "$(cd "$(dirname "$0")/../.." && pwd)/dev/lib/metricsctl.py" "$@"
   ```
   (or equivalent Python-based launcher script).
3. Convert `./metricsctl` at repo root to a deprecation shim:
   - Print `"[DEPRECATED] ./metricsctl is deprecated; use ./dev/bin/metricsctl"` to stderr
   - Then `exec ./dev/bin/metricsctl "$@"`
4. Update docs: `docs/development-handbook.md`, `metrics/README.md`, `design/infra/metrics-ctl-design.md`
   — clarify `./dev/bin/metricsctl` as sole canonical entrypoint.
5. Audit `tests/` for any hard-coded `./metricsctl` paths; update to `./dev/bin/metricsctl`.
6. Confirm `poller_runner._service_unit_content()` already uses `dev/bin/metricsctl` (it does).
7. Confirm static systemd template `metrics/systemd/nightfall-metrics-poller.service` is correct.

**Regression risks and mitigation:**
- Risk: Moving Python to `dev/lib/metricsctl.py` disrupts `sys.path` imports.
  → Mitigation: `dev/bin/metricsctl` wrapper sets up `sys.path` before importing from `dev/lib/`.
- Risk: Existing test fixtures reference `./metricsctl` path.
  → Mitigation: Audit `tests/` for hard-coded paths; update or parametrize before Phase 2 starts.
- Risk: `ensure_venv(Path(__file__))` in new location looks for `.venv` in `dev/lib/`.
  → Mitigation: Phase 1 decouples this; pass `find_repo_root()` result explicitly.
- Risk: Deprecation shim at `./metricsctl` breaks govctl if govctl was not updated in Phase 1.
  → Mitigation: Phase 1 updates govctl first; deprecation shim is safe.

**Acceptance criteria:**
- `./metricsctl` (root shim) prints deprecation warning to stderr, then exits with correct behavior
- `./dev/bin/metricsctl` produces identical output to pre-migration `./metricsctl` for all subcommands
- `./dev/bin/metrics-runner run/status/publish-github` still works
- All govctl `metrics.*` targets pass
- All MCP metrics task tests pass
- No systemd unit regeneration regressions
- Docs updated; no references to `./metricsctl` as primary entrypoint remain

### Phase 3 — Removal & Cleanup

**Goals:**
- Remove `./metricsctl` root-level file after deprecation period has elapsed
- Resolve `metrics-runner` fate: permanent shim with clear deprecation notice, or removal
- Final documentation cleanup, regression gating, and communication

**Non-goals:**
- Do not change `dev/lib/metricsctl.py` implementation
- Do not restructure `metrics/runner/` modules

**Preconditions:**
- Phase 2 complete and validated
- Deprecation notice for `./metricsctl` has been visible for ≥ 2 weeks
- No callers of `./metricsctl` root script remain (confirmed via grep audit)
- Decision made on `metrics-runner` retention vs. removal
- Team sign-off obtained

**Conditions for removal of `metrics-runner`:**
- All operator workflows have been migrated to `metricsctl` subcommands
- Documentation updated to remove all `metrics-runner` usage examples
- Regression tests added covering `run-now`, `status`, and `publish` aliases directly via `metricsctl`
- Team agreement that operator alias semantics (`run`, `publish-github`) are no longer needed

**Work items:**
1. Remove `./metricsctl` from repo root (or move to `dev/archive/metricsctl-deprecated`
   for a single release cycle, then purge).
2. Update `pyproject.toml` and any package manifests that reference `metricsctl` root path.
3. Decision on `metrics-runner`:
   - **Keep as permanent shim:** Add prominent `# DEPRECATED — use metricsctl directly` header
     and version note; no other changes.
   - **Remove:** Grep all docs/scripts for `metrics-runner` references; update all occurrences;
     confirm no CI or operator scripts depend on it.
4. Final docs update: `AGENTS.md`, `README.md`, `docs/development-handbook.md`,
   `metrics/README.md`, `design/infra/metrics-ctl-design.md`.
5. Add CI regression gate: grep check that confirms `./metricsctl` invocations do not appear
   in `govctl-targets.yaml`, `tests/`, or `docs/` (except as historical archive references).
6. Update `CHANGELOG.md` or release notes with breaking-change notice if any external
   consumers reference `./metricsctl`.

**Versioning and communication:**
- Comment on Issue #8 when each phase is complete
- Tag a git ref at Phase 2 completion (before root removal)
- Document breaking change if `./metricsctl` removal affects external integrations

**Acceptance criteria:**
- `./metricsctl` at repo root is absent or clearly marked as removed/archived
- `./dev/bin/metricsctl` is the sole canonical entrypoint, confirmed by full test suite
- `metrics-runner` is absent or contains clear deprecation notice that only calls `dev/bin/metricsctl`
- All docs updated; no stale references to root `./metricsctl` as primary entrypoint
- All existing tests pass
- CI grep gate passes
